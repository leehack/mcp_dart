# MCP 2026-07-28 Transition Guide

For package requirements and application-visible changes when updating from
the stable 2.2 line, start with the
[2.2 to 2.3 migration guide](migration-2.2-to-2.3.md).

`mcp_dart 2.3.0-dev.2` is dual-era: it implements the locked release-candidate
core of the MCP 2026-07-28 specification while retaining the MCP 2025-11-25
feature set and earlier initialization compatibility. Its default
`McpProtocol.stable` profile prefers `server/discover` and stateless request
metadata, then falls back for legacy peers.

## Default client behavior

No protocol option is needed for the default MCP 2026-07-28 negotiation:

```dart
final client = McpClient(
  const Implementation(name: 'my-client', version: '1.0.0'),
);
```

The default behavior is equivalent to
`McpClientOptions(protocol: McpProtocol.stable)`: it enables
`server/discover`, sends the `2026-07-28` stateless request metadata,
and falls back to the legacy `initialize` flow when the peer looks like a
legacy MCP server. On body-only transports such as stdio, a silent discovery
probe is bounded to five seconds before fallback. HTTP transports retain their
normal request timeout because an HTTP timeout represents an unavailable
endpoint rather than evidence of a legacy peer.

Recognized modern protocol errors stay in modern negotiation: the client may
retry a mutually supported stateless version, but it does not reinterpret the
error as permission to run legacy `initialize`. Legacy-only server profiles
identify `server/discover` as unavailable so dual-era clients can fall back.

The protocol era is fixed once a server connection accepts its opener. A
connection that accepts `server/discover` or another stateless request cannot
later run `initialize`, and an initialized legacy connection cannot later send
stateless request metadata. Open a new connection (or a new stdio child
process) to select the other era.

Use the legacy profile explicitly when a deployment must use only MCP
`2025-11-25` and earlier behavior:

```dart
final client = McpClient(
  const Implementation(name: 'my-client', version: '1.0.0'),
  options: const McpClientOptions(protocol: McpProtocol.legacy),
);
```

Use the strict profile for conformance tests or deployments where fallback is
not acceptable:

```dart
final client = McpClient(
  const Implementation(name: 'my-client', version: '1.0.0'),
  options: const McpClientOptions(protocol: McpProtocol.require2026),
);
```

## Default server behavior

`McpServerOptions()` also defaults to the stable profile. Servers advertise
and accept MCP `2026-07-28` stateless requests alongside legacy
versions:

```dart
final server = McpServer(
  const Implementation(name: 'my-server', version: '1.0.0'),
);
```

Use `McpServerOptions(protocol: McpProtocol.legacy)` to advertise only legacy
MCP versions, or `McpServerOptions(protocol: McpProtocol.require2026)` to
reject legacy initialization.

## Stateless identity metadata

MCP `2026-07-28` carries peer identity in stateless metadata:

- Clients normally include `_meta["io.modelcontextprotocol/clientInfo"]` on
  every request, but the field is optional. Servers accept it when omitted and
  reject a value that is present but is not a valid `Implementation` object.
- `McpServer` stamps its configured identity in
  `_meta["io.modelcontextprotocol/serverInfo"]` on successful stateless results
  by default. A valid handler-authored `Implementation` value at that key wins;
  `null` configures an anonymous result by omitting the key, while malformed
  non-null values fail before serialization. `server/discover` no longer emits
  a body `serverInfo` field.
- `DiscoverResult.serverInfo` is nullable and reads result metadata for
  convenience. Missing canonical identity is anonymous; a present malformed or
  `null` canonical identity is rejected. The temporary pre-#3002 body fallback
  ignores malformed legacy identity. Applications must not use self-reported
  identity for security or protocol behavior.

Code that previously dereferenced discovery identity must now handle an
anonymous server:

```dart
final discovery = await client.discoverServer();
final serverInfo = discovery.serverInfo;
if (serverInfo != null) {
  print('${serverInfo.name} ${serverInfo.version}');
}
```

The parser temporarily accepts the pre-spec top-level discovery identity sent by
the published Python beta and older TypeScript peers. Dart never emits that
legacy shape; the fallback can be removed after corrected peer releases are
pinned.

## Profile summary

| Profile | Default? | Client behavior | Server behavior |
| ------- | -------- | --------------- | --------------- |
| `McpProtocol.stable` | Yes | Tries `server/discover`, then falls back to `initialize` | Advertises MCP `2025-11-25` and MCP `2026-07-28` protocol versions |
| `McpProtocol.legacy` | No | Uses legacy `initialize` | Advertises MCP `2025-11-25` and earlier protocol versions |
| `McpProtocol.require2026` | No | Requires MCP `2026-07-28` discovery | Advertises only stateless MCP `2026-07-28` protocol versions |

The `stable` profile name identifies the SDK's default compatibility policy;
it does not label the preferred wire version as final. The constants distinguish
that maturity explicitly: `stableProtocolVersion` is `2025-11-25`, while
`previewProtocolVersion` is `2026-07-28` for this prerelease.
`latestInitializationProtocolVersion` remains `2025-11-25` even after the
stateless protocol becomes stable, so legacy fallback stays wire-correct.
The deprecated `latestProtocolVersion` and `supportedProtocolVersions`
constants retain their mcp_dart 2.2 initialization-era values. Use
`allSupportedProtocolVersions` for the default dual-era profile.

## Low-level overrides

The existing low-level options remain available for advanced callers:

```dart
final client = McpClient(
  const Implementation(name: 'my-client', version: '1.0.0'),
  options: const McpClientOptions(
    protocolVersion: previewProtocolVersion,
    useServerDiscover: true,
  ),
);
```

Prefer the `protocol` profile unless you need to target a specific protocol
version for tests or interoperability debugging.

A supported legacy `protocolVersion` selects `initialize` directly unless
`useServerDiscover` is explicitly enabled. When discovery is forced, the probe
uses the latest stateless version and retains the requested legacy version only
for a genuine legacy fallback.

Use `previewProtocolVersion` to target MCP `2026-07-28` while the
upstream specification is in release-candidate validation. The earlier
`DRAFT-2026-v1` conformance alias is no longer exposed or accepted by the SDK.

## APIs specific to MCP 2026-07-28

The following features use MCP `2026-07-28` behavior in this
branch's default stable profile. Select `McpProtocol.legacy` explicitly when a
caller must avoid MCP 2026-07-28 behavior:

- `server/discover` negotiation and stateless per-request metadata.
- `subscriptions/listen` stateless notification streams, including
  one requested-filter-subset acknowledgment before events and
  `SubscriptionsListenResult` metadata on graceful stream closure. Stateless
  stdio clients restore active subscriptions after an unexpected child exit.
- Direction-specific notifications: stateless clients may send cancellation
  only over stdio, while progress, logging, list/resource updates,
  subscription acknowledgments, and Tasks notifications are server-to-client.
  Known wrong-direction methods are rejected; unknown methods remain available
  to negotiated extensions.
- Multi-result tool/resource/prompt flows such as `input_required`.
- URL-mode `elicitation/create` uses the current draft schema shape with
  `mode`, `message`, and `url`; `elicitationId` and
  `notifications/elicitation/complete` are pre-MCP 2026-07-28 wire shapes.
- MCP Tasks extension flows using `io.modelcontextprotocol/tasks`, with base
  creation seeds, detailed polling/notification state, and per-request
  capability checks. The legacy `TaskStore` adapter is not a modern extension
  persistence layer, although a dual-era server may keep it configured for
  MCP 2025-11-25 clients while implementing the modern extension explicitly.
- Non-object `structuredContent` values via `JsonValue` and broader server
  `outputJsonSchema` shapes.
- Stateless result metadata such as `resultType`, `ttlMs`, and `cacheScope`.

When using the raw `ClientCapabilities.extensions` or
`ServerCapabilities.extensions` maps, extension identifiers follow MCP `_meta`
key naming: they must be prefixed names such as `com.example/feature`. The
official Tasks extension key is exactly `io.modelcontextprotocol/tasks`, and
its settings value must currently be an empty JSON object (`{}`). The typed
`withMcpTasksExtension()` helper applies that shape automatically.

List-change and resource-update notifications are subscription-scoped in MCP
2026-07-28. Send them from the active `subscriptions/listen` handler with
`RequestHandlerExtra.sendSubscriptionNotification`; the global
`sendToolListChanged`, `sendPromptListChanged`, `sendResourceListChanged`, and
`sendResourceUpdated` helpers are legacy-session APIs and are suppressed for
stateless servers.

Server-initiated `ping`, sampling, elicitation, and roots helpers are likewise
legacy-session APIs and throw under stateless MCP. Return
`InputRequiredResult` with embedded input requests from the originating
handler instead. Client transports also reject direct JSON-RPC response sends
after they identify a stateless connection. Stateless HTTP cancellation closes
only the affected request's response stream and never sends a JSON-RPC
cancellation notification.

For deprecated request-scoped logging, prefer
`extra.sendNotification(JsonRpcLoggingMessageNotification(...))` inside the
request handler. The SDK enforces the request's log-level opt-in and routes the
notification on that request's response stream. If using
`sendStatelessLoggingMessage`, pass both `requestMeta: extra.meta` and
`requestId: extra.requestId`; the ID is required for Streamable HTTP routing.

MCP 2026-07-28 retains request-scoped logging for compatibility but marks it
deprecated. New implementations should use `stderr` with stdio or OpenTelemetry
for structured observability; the current examples use progress and
`subscriptions/listen` instead.

The MCP 2026-07-28 multi-round callback APIs are additive. Existing
`registerTool`, `registerPrompt`, `registerResource`, and
`registerResourceTemplate` callbacks retain their `mcp_dart 2.2.2` result
shapes. Use the corresponding stateless registration only when a callback may
return `InputRequiredResult`. Tools also use stateless registration when their
output schema has a non-object root:

| Existing registration | Additive MCP 2026-07-28 registration |
| --- | --- |
| `registerTool` | `registerStatelessTool` |
| `registerPrompt` | `registerStatelessPrompt` |
| `registerResource` | `registerStatelessResource` |
| `registerResourceTemplate` | `registerStatelessResourceTemplate` |

For non-object tool results, keep the existing object-root APIs for
initialization-era callers and use the explicitly named MCP 2026-07-28 APIs:

```dart
server.registerStatelessTool(
  'array-result',
  outputJsonSchema: JsonSchema.array(items: JsonSchema.string()),
  callback: (args, extra) {
    return CallToolResult.fromStructuredArray(['alpha', 'beta']);
  },
);

final result = await client.callTool(
  const CallToolRequest(name: 'array-result'),
);
final items = result.structuredContentJson?.asArray;
```

`registerStatelessTool()` returns a `RegisteredStatelessTool`. Use its
`outputJsonSchema` and `statelessCallback` getters to inspect the complete
registration, and `updateStateless()` to update a non-object schema without
narrowing it to the legacy object-rooted `outputSchema` view.

The preview API surface may still change before the official spec release.
Use `McpProtocol.legacy` when an application cannot accept preview behavior;
otherwise the default `McpProtocol.stable` profile exercises MCP 2026-07-28
with legacy fallback.

## Release status

Published SDK and CLI prereleases provide the MCP `2026-07-28` preview channel.
The standalone install scripts track stable releases; activate an explicit CLI
prerelease when testing this protocol. Projects created by the dev.2 CLI use the
matching dev.2 SDK dependency.

[PR #306](https://github.com/leehack/mcp_dart/pull/306) merged the MCP
2026-07-28 implementation line into `main`. The superseded RC PR is closed,
while `dev/2026-07-28-rc` remains a read-only archive for links embedded in the
earlier dev.0 and dev.1 packages.

CI runs official MCP 2025-11-25 and MCP 2026-07-28 conformance plus pinned
spec-example audits, bidirectional published TypeScript beta.5 interop,
published Python server interop with its exact client gap, real-browser
transport tests, a Flutter Web service integration in Chrome, and widget
tests. See
the [interoperability guide](interoperability.md#running-interop-checks-locally)
for local commands and the
[day-0 release runbook](mcp-2026-07-28-release-runbook.md) for the final tag,
metadata, publish, and public-verification gates.
